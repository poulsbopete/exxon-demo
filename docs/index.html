The Problem
Root Causes

What the Teams Say Today

  • 🔴
    Logging & tags — Logging pipelines are not well-defined. Service delivery organizations (SDOs) do not effectively leverage tags across metrics, logs, monitors, or events. Some tags are thought to be "product specific" (e.g. service, often explained as mainly for APM).
  • 🔴
    Tagging strategy vs. implementation — ExxonMobil has an internal tagging strategy, but it is up to SDOs to implement it — and not all SDOs understand or apply it consistently.
  • 🔴
    OpenTelemetry — OpenTelemetry is not heavily utilized at ExxonMobil; most organizations focus on Datadog Agent, APM, or logs. Elastic Serverless accepts OTLP natively — if Azure or other sources can emit OTLP, it can be collected easily in one place.
  • 🔴
    Network invisible to app teams — OpenNMS sees the Cisco SNMP linkDown trap. Datadog does not. A 3-flap circuit at Houston Refinery takes 45 minutes to correlate to the api-gateway errors it causes.
  • 🔴
    "Jitter DNS" — an internal term for a known WAN problem — MPLS jitter on the Midland circuit causes Azure AD auth timeouts, Event ID 4625 storms, account lockouts. WAN team, AD team, desktop team, and security team never see the same dashboard.

Challenge 2 — SNMP + Network Correlation

Network Events & APM in the Same Platform

The Scenario

  • 🔴
    Cisco circuit flap — GigabitEthernet0/47 on cisco-sw-houston-01 flaps 3× in 5 minutes
  • ⚠️
    OpenNMS sees it — SNMP linkDown trap received. App team in Datadog? Zero visibility.
  • 🔴
    api-gateway errors spike — upstream connection resets from Houston-Refinery-Campus correlate directly — but nobody connects the dots
  • ✅
    Elastic correlation — SNMP trap + APM error, same platform, one ES|QL query, shared network.site join key
"What Cisco switch is flapping at Houston and what business service does it support?"

The ES|QL Queries

-- 1. Find SNMP trap events
FROM logs*
| WHERE snmp.trap.type IS NOT NULL
| KEEP @timestamp, device.hostname,
  snmp.trap.type, network.site
| SORT @timestamp DESC
-- 2. Cross-correlate: network + APM
FROM logs*
| WHERE (snmp.trap.type IS NOT NULL
  OR service.name == "api-gateway")
| STATS event_count = COUNT(*)
  BY network.site, service.name
| SORT event_count DESC

Focus: Monitoring & Alerting

SLO & Burn Rate — How We Use It

In Datadog the relationship between SLOs and monitors is bidirectional and ambiguous; Elastic is more deliberate, with clear burn rate windows and action groups.

  • 📐
    How is burn rate measured? — We want to understand how burn rate is determined in Elastic (windows, thresholds) so we can align with how monitoring should be set up.
  • 🎯
    Tactical alerting — Use burn rate on an SLO with a group-by field (e.g. Site code) to detect multiple sites with the same issue — a "storm" — and alert operations.
  • 📢
    Management alerting — Use SLO violations to alert management organizations, so leadership sees when commitments are at risk.

Elastic offers more options for burn rate windows, action groups, and a clear SLO → rule relationship.
